リサーチ結果とRichard P. Gabriel氏の「4層モデル」に基づき、Common Lispコンパイラ実装の\*\*最初のマイルストーン（Milestone 1）\*\*を提案します。  
このマイルストーンのゴールは、Gabriel氏が提唱する\*\*「第1層：カーネル言語 (Kernel Language)」の確立\*\*です。  
目的は「動的な再定義を必要としない、単純で実装しやすい基盤」をWasm GC上に構築することです。この段階ではコンパイラ（Lisp to Wasm）そのものよりも、\*\*Lispのデータ構造と基本操作を実行できるランタイム（Wasmモジュール）\*\*の完成を目指します。

### ---

**マイルストーン 1：カーネル・ランタイムの構築 (The Kernel Runtime)**

目標: Wasm GCを利用したLispデータ構造の定義と、それらを操作する不変（Immutable/Static）なプリミティブ関数の実装。  
成果物: kernel.wasm (手書きWATまたはRust/Zig等から生成) および JSホスト側の最小限のブリッジ。

#### **1\. データ構造の型定義 (Wasm GC Type Definitions)**

Wasmの型システム（struct/array/i31ref）を使用して、Lispのタグ（Tag）を模倣せず、エンジンネイティブな型として定義します 1。

* **Universal Type:**  
  * 全てのLispオブジェクトのトップタイプとして (type $any (ref null any)) または eqref を定義。  
* **Cons Cell (第1層の核心):**  
  * car, cdr を持つMutableな構造体。  
  * コード スニペット  
    (type $cons (struct (field $car (mut $any)) (field $cdr (mut $any))))

* **Symbol (カーネル識別子):**  
  * Gabriel氏の言う「再定義」は上位層の話ですが、カーネル層ではそのための「スロット」を用意します。  
  * コード スニペット  
    (type $symbol (struct  
      (field $value (mut $any))    ;; 値セル (Special Variable用) \[2, 3\]  
      (field $function (mut $any)) ;; 関数セル  
      (field $name (ref string))   ;; 名前 (Wasm stringref or externref)  
      (field $plist (mut $any))    ;; プロパティリスト  
      (field $package (mut $any))  
    ))

* **Fixnum:**  
  * 構造体ではなく i31ref をそのまま使用し、アロケーションコストをゼロにします 4。

#### **2\. カーネル・プリミティブの実装 (Kernel Primitives)**

これらは「再定義不可能」な固定機能として kernel.wasm からエクスポートされます。

* **Allocation:** cons, make-symbol  
  * JS側から呼び出してオブジェクトをヒープに確保する関数。  
* **Accessors:** car, cdr, symbol-value, symbol-function  
  * Wasmの struct.get / struct.set 命令への薄いラッパー。  
  * 型チェック（ref.cast）を含み、Consでないものに car をした場合はトラップ（例外）させます。  
* **Predicates:** consp, symbolp, fixnump, eq  
  * eq は ref.eq 命令にマッピングされます。  
  * fixnump は ref.test (ref i31) を使用します。  
* **Arithmetic (Fixnum only):**  
  * \+, \-, \=, \<  
  * i31ref 同士の演算のみをサポート。オーバーフロー時はとりあえずトラップするか、i64への昇格（Bignum準備）のスタブを残します。

#### **3\. 最小限の環境 (Environment / Epilinguistic)**

第1層をテストするための「足場」をJavaScript側で作ります。

* **Bootstrapper (JS):**  
  * WebAssembly.instantiate で kernel.wasm をロード。  
  * NIL オブジェクトと T オブジェクトを初期化時に生成し、Globalとして保持。  
* **Reader (JS):**  
  * 簡易的なS式パーサーをJSで実装。  
  * 文字列 "(cons 1 2)" をパースし、Wasmのエクスポート関数 cons や i31ref 生成を呼び出して、Wasmヒープ上に構成する。  
* **Printer (JS):**  
  * Wasm GCオブジェクト（externref としてJSに見える）を受け取り、型判別して (1. 2\) のようにコンソール表示する機能。

### ---

**このマイルストーンの検証コード例**

このフェーズが完了すると、以下のことが可能になります（JSコンソール上での操作イメージ）：

JavaScript

// JS側でWasmのKernelを操作する  
const kernel \= await loadLispKernel();

// 1\. Symbolの生成 (Interningはまだ簡易的でよい)  
const symA \= kernel.intern("A");  
const symB \= kernel.intern("B");

// 2\. Consセルの作成: (cons 'A 'B)  
const cell \= kernel.exports.cons(symA, symB);

// 3\. アクセス: (car cell) \-\> 'A'  
const carVal \= kernel.exports.car(cell);  
console.log(kernel.printer(carVal)); // Output: A

// 4\. Fixnum演算: (+ 10 20\)  
const result \= kernel.exports.add(  
  kernel.exports.to\_fixnum(10),  
  kernel.exports.to\_fixnum(20)  
);  
console.log(kernel.printer(result)); // Output: 30

### **次のステップへの接続**

この「カーネル」が安定して動作することを確認した後、次のマイルストーンで「第2層：言語学的層」である **Wasm内インタプリタ (Eval)** の実装に進みます。そこではじめて、lambda や if といった制御構造が導入されます。
